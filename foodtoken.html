  • Virtual Food Token System




    The Problem

    In our annual college fest, volunteers were given food tokens in return for their work, which they could redeem at the cafeteria. Now these were small plastic chips, which could not be traced back to any legitmate source. Students in charge could also give their volunteer friends, extra tokens. Or in some cases, keep extra tokens for themselves and use them.
    This meant that the college was having to pay for more food tokens than nessesary

  • Our Solution

    We decided to make an online coupon transferring system. Every volunteer and event organiser would get an user account, which in return held a balance of tokens. This user account would be linked to their college ID for identification.
    These coupons could be transferred virtually from one account to another, given that it was allowed and did not break any rules.
    Every time someone wanted to use a token, they could simply go up to the cafetaria, and on the click of a button on the web app, their coupon would get converted into a unique QR code. This QR code could then be scanned by the people present there. If the transaction was succesful, the balance of the user would be deducted and they would then get food.
    The scanning part was done by the a different user type on our web app meant only for the people managing the cafeteria and food. Only they would be able to see this feature.

  • User accounts, Restrictions and Logging


    There were 7 different user accounts in reality. Each level would be allowed to give coupons to the level below it, given that both were off the same event (for event heads) or of the same category (for category heads, as 1 category could hold multiple events). But for simplicity in explanation, Lets club them into these 3 types:
    1. Event Organisers
    2. Cafetaria and food managers (for scanning)
    3. Volunteers

    Again, please note that this is over simplified. In reality, there was a much more complicated system of interactions between users.
    The Event Organisers had an option to give 1 coupon per day to a volunteer in their event only. Every user and volunteer was mapped to an event in the database, so Event organisers of an event could not give tokens to volunteers of a different event.
    There was also a strict restriction of giving only 1 token per day to the volunteer. Once the token was given, the volunteers name would automatically disappear form the Event Organisers list till the end of the day. Note, that there can be multiple Event Organisers for each event, thus this concurency was taken care of.
    The Event organiser too, had a restriction of using only 1 token per day. They were automatically given 1 token per day by the system for personal use.

    Each transaction, or act of giving or redeeming a coupon was logged along with its timestamp, and the IDs of both users involved to provide proof and help in tallying the bill against the cafetarias.

  • Transactions


    The transaction was done in a process like this -
    1. The user would first click on "Generate QR Code". Here the system would first check if the user had enough balance in the database. If yes, then QR code would be generated.
    2. Now the cafetaria manager would scan QR code. Here, the managers phone would extract the user ID and number of coupons requested from QR code. It would then perform another check to see if the user account had enough coupons. This extra step was to prevent users from using screenshots and availing unlimited coupons.
    3. If the scanner detected a valid amount of tokens in the users account, it would then deduct the balance. This was done in a single transaction to avoid data mismatch on failure.
    4. Finally the user would be given food
    This whole process would happen in under a second!

  • Implementation


    The web app ended up going live and attracting 1000+ users. We did 7000+ transactions worth more than 1.6 Lakh Rupees!
    We used Amazon EC2 instances to host this, along with a MySQL database and node + Express for the backend. The flow of coupons is shown in a much simpler than in real life. It was much more complicated than this due to different organisations and different rules for each of them, and many different levels of fest managers. Sadly, all that can be quite confusing and is out of the scope of this article.

  • End notes

    This was a really fun and challenging project to do. The cherry on top was definetly it being used on such a large scale in my college. I learnt a lot about designing these systems and the workflows around them. We had to look into security concerns, user interactions, encrypting data, etc. Overall, it's definetly one of my favioriate projects.
